Start Experimenting with Bebop

Clone the project, let your AI assistant set it up, and start creating your own design experiments on top of the Copilot Shell.

Prerequisites
Node.js — v18 or later. Download here
A code editor with an AI assistant — e.g. VS Code with GitHub Copilot, or Claude Code
Getting started
1
Get the project running Open the IDE and AI assistant of your choice — VS Code with Copilot, Claude Code, etc. Clone the project:
"Clone this repository: https://github.com/m365-core/bebop-playground.git"
Set up and preview:
"Install dependencies, build all previews, start the local server, and open Bebop Playground in the browser."
This should open the Bebop Playground in your browser. Verify your setup: Explore and play around with the Shell and Design System Tokens to ensure your local environment has been set up correctly. You can verify interactions against the GitHub repo.
2
Create your first experiment Every experiment starts as a copy of the Copilot Shell, so you get the prototype as your starting point. Your changes layer on top without affecting the original. Ask the AI:
"Create a new flow exploration called 'myExperiment' based on the shell. Build it and open it in the browser."
You can open your experiment directly from the link the AI gives you, or find it on the Prototypes page in the catalog. Verify: Your experiment should look identical to the Copilot Shell — that's expected. It loads and works independently from the original.
3
Make your first change Start with something small to confirm the AI is editing the right file. Ask the AI:
"In myExperiment, change the greeting heading to 'Hello World'. Rebuild and open it in the browser."
Verify: You should see your new heading. If it doesn't appear, ask the AI which file it edited — make sure it's your experiment file.
Building your design
4
Build your design Now you can start making real changes. Describe what you want in plain language — the AI will edit your experiment file, rebuild, and show you the result. Using existing components:
"Add a banner below the chat input showing connected app icons for M365, Jira, and GitHub."
Using Figma designs: Paste a Figma link directly into chat. The AI extracts colors, spacing, and layout from the design.
"Here's my Figma design: [paste URL]. Implement this as a new overlay in myExperiment."
Iterating: Describe what's wrong or what to change next. The AI edits and rebuilds — you just refresh.
"The modal is too wide, make it 480px max. Also add more padding around the close button."
Getting better results
5
Tips for better outputs A few habits that help the AI give you better results, faster. Scope your changes to an experiment When starting a new experiment, ask the AI to only make changes to that file. You only need to say this once — the AI will respect it unless you ask otherwise.
"Only make changes to myExperiment.ts — don't touch any other files."
Be specific and reference existing components The more detail you give, the closer the first result. Browse the Design System Tokens page to see what's already built, then reference them by name.
"Use the same card style from the connectors flow for my source picker."
Contribute new components back If you've created a useful primitive or component in your experiment, you can add it back to the design system so that the AI is able to reference it easily.
"I've built a notification toast in myExperiment. Add it as a new component to the design system with a token file, preview, and catalog entry."
Icons 94 icons (Fluent UI + app logos) are included. You can also add new ones — download an icon and ask the AI to use it.
"What icons are available?"
Check the browser console If something looks broken, open DevTools (F12) → Console. JS errors in preview generators fail silently. Paste any errors back into chat.
Contributing back
6
Contribute to the community Built something useful? Share it back so others can use it too. The AI handles branching, committing, and PR creation. Ask the AI:
"Commit my changes and open a pull request to the main repo with a description of what I built."
The maintainer reviews your PR, may suggest changes, and merges it. Your contribution will then be available to everyone using the playground.
Something's not working? Ask your AI to raise an issue on GitHub
